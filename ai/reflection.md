# 最终反思

## 1. 你使用了哪些 AI 工具或模型？

我使用了 Codex / GPT-5 辅助阅读需求、规划结构、实现代码和检查测试。

## 2. 哪个提示词最有用？为什么？

最有用的是要求实现完整计划的提示词，因为它明确限定了语言、项目结构、功能范围和不使用 Node.js 的约束。

## 3. 哪个 AI 生成的建议是错误、不完整或误导性的？

最需要人工判断的是 JSON 持久化方案、排行榜规则和公开检索边界。通用 JSON 解析器会让项目复杂化，因此最终只实现适合本项目固定数据结构的读写逻辑；排行榜同位排序则根据用户决策改为等级、胜率、对战次数和 ID 的稳定顺序；公开玩家搜索只允许检索玩家身份字段，不暴露管理员和教练超级账户。

## 4. 你如何检查 AI 生成的代码是否正确？

我通过阅读类职责、运行 `javac` 编译、执行控制台菜单、检查查询结果、保存加载结果、连续登录结果、错误输入提示和外部 JSON 更新结果来验证。

## 5. 哪些 bug 是你自己修复的，而不是要求 AI 修复的？

在最终检查中，我会根据编译错误和菜单运行结果手动修复格式、输入处理和数据一致性问题。连续登录评测时发现原数据集只有一个管理员账号，无法覆盖多个管理员连续登录，因此补充了 `coach/coach123` 测试账号。公开查询调试时发现玩家 ID 片段不能触发候选列表，因此补充了模糊候选和精确直达逻辑。

## 6. 使用 AI 后，你对哪个 Java 概念理解得更好？

我对接口和服务分层的关系理解更清楚，尤其是如何让搜索、认证和持久化分别承担不同职责。

## 7. 你对哪个 Java 概念仍然不确定？

我对更复杂的通用 JSON 解析和异常体系设计仍然不完全确定，因此本项目选择了较简单、可解释的方案。

## 8. AI 使项目更容易、更难，还是两者都有？解释。

两者都有。AI 让规划和样例代码更快，但也需要我判断哪些建议适合课程要求，哪些会引入不必要复杂度。

## 9. 最终项目的哪些部分主要是你写的？

菜单选择、功能验证、文档判断、测试结果确认和最终提交决策主要由我负责。

## 10. 哪些部分主要是 AI 生成或得到 AI 大力协助的？

类结构规划、服务类初稿、文档初稿和测试用例初稿得到了 AI 的大力协助。
