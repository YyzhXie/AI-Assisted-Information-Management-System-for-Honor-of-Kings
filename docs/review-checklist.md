# 评审检查清单

## 功能覆盖

- 玩家查询：已实现，支持 ID、用户名和昵称关键字。
- 战队概览：已实现，显示成员、平均等级、总场次、胜率和顶尖玩家。
- 英雄详情：已实现，显示属性、装备、拥有者和推荐装备。
- 装备统计：已实现，按综合实力自定义分数排序。
- 对战历史：已实现，支持玩家和战队最近 N 场。
- 对战模拟：已实现，支持控制台和 Swing GUI 生成 5v5 预览战报；少于 5 名可参赛成员拒绝模拟，超过 5 人随机抽 5 人，不修改真实记录。
- 排行榜：已实现，支持胜率、等级、对战次数和综合实力。
- 数据管理：已实现管理员对玩家、英雄、装备、战队、对战记录的添加、编辑、删除。
- GUI 管理员表单：玩家、英雄、装备、战队、对战记录已实现自动 ID、必填项、引用存在性校验、覆盖确认和删除确认；战队支持先创建空队。
- 身份验证：已实现管理员和玩家登录、登出。
- 文件 I/O：已实现 JSON 保存和加载。
- 连续登录稳定性：已通过多个玩家、多个管理员、混合角色和错误密码场景。
- 交互输入鲁棒性：已通过非数字、越界、空输入、错误日期、错误枚举和不存在记录场景。
- 长线运营更新：已通过外部 JSON 添加 `T004` 战队和 `E021` 装备。
- 输出准确性：已逐项核对玩家、战队、英雄、装备统计、对战历史和排行榜输出。
- 派生数据一致性：已通过新增/删除玩家、英雄、装备后的战队统计、英雄拥有者和推荐装备同步测试。
- 综合实力公式：已用 `ComprehensiveRankingTest` 验证玩家和装备综合实力计算。
- 对战模拟稳定性：已用 `CombatSimulatorTest` 验证固定随机种子、非法战队输入、少于 5 人拒绝、超过 5 人抽 5 人和不落库规则。
- 排行榜同位排序：已按“等级、胜率、对战次数、ID”决策更新；单局比赛不设置平局。

## Java 概念检查

- 继承、接口、多态、封装、集合、异常处理、文件 I/O、枚举均有对应代码。
- 主方法只负责启动，业务逻辑分散在服务类中。
- 初始数据满足最低数量要求。

## 风险检查

- JSON 解析器只面向本项目固定结构，已在 README 和设计文档中说明。
- 控制台输入已通过 `InputHelper` 处理非法数字输入。
- 当前认证状态为单用户控制台会话，适合本项目；若改为并发系统，需要引入会话对象。
- 外部 JSON 更新需要维护 ID 关系；装备参与推荐时必须加入英雄兼容装备列表，战队成员变更时应同步玩家 `teamId`。
- 战队历史总对战数和胜率来自对战记录；仅添加或删除玩家不会改写历史对战统计，这是当前数据模型的预期行为。
- 对战模拟只作为预览报告，不会自动生成对战记录；如果以后要落库，需要重新设计玩家胜负场和历史记录的一致性策略。
